I've searched in the forum about "br-lan: received packet on eth0.1 with own address as source address" and found some things, but could not solve my problem.

I have more than one wireless router in my home network, one of them is using openwrt and i'm using it as DHCP and DNS.
From time to time I'm getting this "br-lan: received packet on eth0.1 with own address as source address" and in the same moment that router freezes (do not respond pings, dns, nothing).
Searching in the forum, I've found something saying about turning off the bridge. I did it, the message doesn't appear again and the router doesn't freezes anymore. BUT... the bridge that I've turned off is the bridge between eth0.1 (wired network) and my wireless networks.

Is there a way to stop this?
Or a way to not use "bridge interfaces" and put my wired and wireless networks together?

My router (working as AP) is a Netgear WNDR 3700 v1, with "OpenWrt Attitude Adjustment 12.09 / LuCI 0.11.1 Release (0.11.1)"
The other routers are:
- one Cisco RV320 working as the internet router
- one Netgear R6300 with original firmware, working as AP
- one Apple Airport Extreme, working as AP

Before I was using a VM with openwrt as the internet router, the netgear 3700 and the netgear 6300 as APs. I think the problem existed, but the netgear 3700 was being used only as AP, so I could not see the problem happening.

anyone?

Ok...
I didn't understand what this does, actually, but reading the "Dumb AP" in the wiki and seeing (through Microsoft Network Monitor 3.4) "Igmp: IGMP membership query" (source: 0.0.0.0 , dest: 224.0.0.1) I configured the /sys/devices/virtual/net/br-lan/bridge/multicast_snooping to 0.
The message was appearing every 2 minutes and after this change, it has not appeared yet (20 minutes).
LOOKS like the problem is solved, but... what I lose with multicast_snooping = 0 ?
