I'm pretty sure this can be done.  My first question is "Should I do this using routing tables, or using the firewall tables, or both?"

On the client, I could have a routing table kinda like:

Kernel IP routing table
Destination     Gateway         Genmask         Flags Metric Ref    Use Iface
10.8.0.5        0.0.0.0         255.255.255.255 UH    0      0        0 tun0  The VPN
192.168.160.0   0.0.0.0         255.255.255.0   U     0      0        0 br0   local LAN ports
71.114.211.0    0.0.0.0         255.255.255.0   U     0      0        0 vlan1 need this?
10.8.0.0        10.8.0.5        255.255.255.0   UG    0      0        0 tun0  need this?
192.168.9.0     10.8.0.5        255.255.255.0   UG    0      0        0 tun0  remote LAN
75.75.75.75     71.114.211.1    0.0.0.0         UG    0      0        0 vlan1 VPN server IP

If I've got it right, I'll be able to send packets to 75.75.75.75 via my ISP gateway at 71.114.211.1, and the VPN tunnel will go through that connection.

But if I remove the lines marked "need this?" I think that I will ONLY be able to send packets to 75.75.75.75, and there won't be any open route to the internet.

The server needs to always be listening for an incoming connection, so I might have to do everything with IP tables on the server.  And, if I have to learn more about IPtables to make the server work, maybe I should just use IPtables on the client, as well.

But I think it'll be the same kind of thing -- just done with IPTables instead of the routing table.

Thanks for pointing out that it was a pretty vague post- does it sound like this is a workable direction?

The more I think about it, the more I think that IPTables is the way to go, because of the server issues.
