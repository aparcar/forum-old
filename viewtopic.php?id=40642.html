Hi Omiez,

The WR841 will work fine as an AP, and you can connect via wireless. If all you need is standard IPV4 internet access, I think it will
do this nicely. But, then the standard firmware will also do this.

I bought two of these, and I have found several restrictions with current Openwrt. Channels 12 and 13 are not available in Europe, IPV6 does not work very well, and some other things, like LAN file/device sharing do not work.

If you need to bridge, it will do this as well. I used my pair as a WDS link between two buildings, one set as AP + WDS and the other as CLIENT + WDS. THis worked (with the above restrictions) Again, the tp-link firmware worked OK too.

When I restored the firmware from Openwrt to TPlink, I bricked one router. I do not know the exact cause, I used the identical procedure for both.

My findings using TL-WR841ND, is that bridging between wlan and switch (Ports 1-4) is not letting through neighbor discovery protocol, so hosts on wifi cannot talk to hosts on wired lan. Possibly this is a generic bug because other multicasts do not get through also, such as USB memory sharing. I have not tested ALL possible configs, but used WDS bridging, LAN A -> 841ND -> WIFI -> 841 ND -> LAN B, and I could not get my computers in different buildings to talk to each other. I used A.A. beta2 image + kmod-ipv6 . I tested many other options, but nothing worked. It seems with the changes in multicast routing, the bridge code in linux is NOT transparent any more, and that if you want to use multicast addresses you have to register them. Probably someone has a solution to this, but to me it just looks *broken* right now. A lot has happened since I used Linux bridging, back in 2002-2003.....

I verified this is the problem, since adding a "static neighbor" at each end nodes could talk. (this is the IPV6 "ARP" replacement...)

Other routers with Openwrt *might* behave differently, but my investigations indicate this is generic....
