I noticed that sometimes, sysupgrade may drop the current ssh session when it is in the process of killing off running daemons prior to flashing the binary file. This may result with a bricked device because flashing is interrupted when sysupgrade is killed along with the terminated ssh session.

Fortunately I'm using a Netgear device with failsafe recovery, but is there a way to prevent the ssh session from being dropped by sysupgrade's daemon killing process?

Unfortunately that is not the case. sysupgrade (or any other unix shell process) will stop running the moment any ssh shell session is terminated.

This is correct, if you close ssh session yourself. If connection terminates unexpectedly, running programs continue to run. Try yourself:
ssh to router using putty, ping www.google.cz, unplug cable, close putty. Now open putty and ssh to your router again. Run ps and you will see ping www.google.cz still running

Btw I have sysupgraded about 20 devices multiple times, but if I experienced a brick, it had been because of my own stupidity I think there's no need to worry, it is taken care of, dropbear is killed by reboot, not during sysupgrade.
