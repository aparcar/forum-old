If you do something like this you should understand your threat model.

If you encrypt that setting you need to update the appropriate boot/init scripts to not pull plain text from the file but to decrypt some string.  Then you'll need to store a key and pull that from somewhere -- but unless you password protect the key (which would prevent auto-enable of wireless on boot) it will need to be stored in plain text somewhere.

In other words... secure root access to the router and you don't need to worry about that plain text config.  If you're worrying about things like this then hopefully you've at least disabled the serial console... (and if you do, understand implications with respect to debricking your router)

I say this in the best possible way, and in a friendly way, as we are all just trying to help each other out , so just in case you didn't know........

base64 is for encoding
md5 is for hashing

Really, the only reason why the contents of a router, serverOS, application or database would be encrypted would be for a compliance reason (Extra Security). As long as good password complexity is used, and patches/updates are being done regularly, there shouldn't be a reason for encryption of data at rest. That's not to say one can't or shouldn't do it, but that's the basic rule of thumb.
