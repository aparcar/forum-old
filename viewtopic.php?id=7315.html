Working with RC5 on a WRT54GL, I can't get dhcp-option 66 to work.
Even if all other settings from dnsmasq.conf get processed, this one doesn't.
So I decided to call dnsmasq manually, specifying this option as in the following cmd:
  dnsmasq -K -F 10.41.10.20,10.41.10.40,255.255.0.0,12h -I vlan1 -O 66,10.41.10.10
or else
  dnsmasq -K -F 10.41.10.20,10.41.10.40,255.255.0.0,12h -I vlan1 --dhcp-option=66,10.41.10.10
but even this way, my WRT54GL won't send out this information when a client performs a dhcp-request. That is what ethereal tells me.

Anybody an idea, is this a bug, a restriction somewhere from dnsmasq, or just something I didn't see?

Thanks already.

Hi peng,

thanks for your answer.
I'm using thomson IP-phones, and they can use both options 66 & 150 to the same end.

Nevertheless, this topic has found a solution in a different way than I expected.
The answer lay in my ignorence of the dhcp-protocol, but I didn't find it on the web...
so I hope by posting it, I can help some others.

In fact this option 66 is working, as are all other options probably!
But what I didn't know was that in order to see this information sent out, you need a dhcp request that contains this option.
Therefore, tracing with ethereal on a computer will never show this option 66 because the computer never asks for it.
But it does show up on the communication with the ip-phones.

Hope this information finds its way to others in a similar situation.

S
