Somehow, one end of the "tunnel" will need to communicate its IP address to the other. Generally, you will need the router to report its current IP address to a server that it can identify by a fixed IP address, or through DNS. The other option is if the "far end" of the tunnel is at a known address, then  you can watch the tunnel and if it goes down, reconnect from the router.

DynDNS is one way to do that. Having the router "phone home" to a server at a known or DNS address is another. For example, I have one router I maintain running a cron job that HTTP GETs a URL on a server that I control and I can get its IP address from the request (either from the logs, or from a script on the server at the target URL).

If both ends of the tunnel can change IP addresses, you could also hope that they never change at the same time and write some kind of fancy script that knows the last IP address of the far end and connects to that any time the near end changes IP addresses. This isn't an easy thing to make reliable.

IPv6 might be helpful, but there are a lot of dependencies to get right, as well as needing an ISP or IPv4<->IPv6 gateway that support it in a reasonable way.
