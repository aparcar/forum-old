"Please don't believe in this stealth nonesense. If anything, "stealth" tells an attacker that there *is* a firewall dropping packets - it's like holding up a sign stating "nothing to see here". Also dropping ICMP echo replies gains you zero security, if at all it just disrupts services. E.g. HE.net does not let you establich an IPv6 tunnel if you do not allow pings."

see https://forum.openwrt.org/viewtopic.php … 77#p137277



"In general please don't buy too much into the stealth nonesense. It merely violates RFCs and does not prevent anyone from "hacking your router" or perform port scans. Modern scan engines are either massively parallel or distributed - they do not care whether you DROP packages."

see https://forum.openwrt.org/viewtopic.php … 57#p250957

userr wrote:

Stealth although does not prevent attack, it signals attacker there is no device connected

No, it really doesn't. Dropping all packets (what grc unashamedly calls "stealth") is a distinctly different reaction than the network node up the line signaling back that there is no device connected. Or in other words: You don't answer, but one can see that you are not answering. Whether this encourages or discourages any behaviour is purely up to speculation.

"You can DROP, but you cannot hide."

if you want to know some of the reasons net.netfilter.nf_conntrack_skip_filter was removed.

If you want CC to pass that test change in firewall wan from reject to drop and disable the rule that allow the ping (ipv4 one).
If you want CC to pass other tests you will have to disable other rules. (You kinda have to know what you need and what you don't and that depends on how you get internet.)
If NAT leak is the issue activate drop invalid in firewall.

How much doing this will help you? Depends...
