As far as I know Chaos Calmer will limit the speed on wireless to 130 Mbps (or something like that, not sure how exactly it happens in 1043 case, main thing is that the channel width will be limited to 20 Mhz) if there are other wireless networks around.

You can try to add the following line:

option noscan '1'

to

config wifi-device 'radio0'

area in 

/etc/config/wireless

and see if it changes something.
BB had an option in Luci for this, in CC that option is gone.

( 2 links in case you might need them:
https://wiki.openwrt.org/doc/howto/user.beginner.cli - editing files section
https://wiki.openwrt.org/doc/uci/wireless )

In Luci - Network - Interfaces - Wireless - General Setup - Operating frenquencies make sure that:
- Mode is set to N
- Width is set to 40 Mhz (might default to 20 Mhz)

Do a speed test on lan (copy one file between 2 pc connected on lan and check the top speed and how long it took) followed by one in wlan (copy the same file between 2 pc connected on wlan and check the top speed and how long it took) (i'd do one with 1 pc connected to lan and the other one connected to wlan trying trying to copy from lan to wlan and the reverse if the test with both on wlan gives low speed results) just to remove the possibility of a low internet speed caused by the isp.

Does the slow wifi problem go away if you return to the stock firmware?

The high ping on wireless is not really normal. You supposed to see similar speed results on lan and wireless (21.75 Mbps is considerable less compared to what your router should be able to handle on wireless) and maybe a bit worst latency (something like max 5-10 more ms) on wireless (to be honest 22 Mbps should put no stress, I start to see increase in latency when I pass 65-70 Mbps).

What values OpenWRT shows in Signal/Noise for the Associated Stations?

Can you post what 

cat  /etc/config/wireless

shows (edit what it shows at option ssid and option key for security reasons )?

I don't own an 1043 so my help will be limited because I can't do any tests on such router .

See how it does with stock firmware.

If with the stock firmware you see the same behaviour:
1) i'd check the client(s) drivers, test with a client that it's known to work for sure
2) you had the bad luck to get 2 routers with issues on wireless (might be an entire lot with problems, assuming you bought them in same time from same shop)

If with stock firmware the problem is gone:
1) there is a regression with ath9k that affects AR9103 (supposed to see more posts complaining about this problem, there is an old ticket about this to be honest: https://dev.openwrt.org/ticket/19669 )

For "boot" firmware execute the command below, and flash with 'tplink.bin'

dd if=orig.bin of=tplink.bin skip=257 bs=512
