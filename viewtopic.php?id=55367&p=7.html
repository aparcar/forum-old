160 Mhz is not possible with the R8000, not with stock firmware, not with OpenWrt.

80Mhz on the other hand should be working fine. There is one requirement which we unfortunately can not detect in our driver regarding the 5Ghz, and that is that the first radio is for the higher bands (148 or something else, depending on country) and the second 5Ghz device (radio2) should be used for the lower bands (48, or something like that and lower). However this requirement does not have impact on functionality. If you use the wrong channel it should still work (with 80 Mhz bands) and only impact performance.

As soon as I have access to an R8000 I will give it a try with country code UK and see what happens. You could try US, I've tried that country code in the past locally and it works fine for me. Don't forget to reboot your AP after changing the country code.

If you get them up and running the wireless should perform fine. I've gotten speeds of at least 600Mbit/s (on short range) for long periods of time.

Spent some time today on this issue and there is indeed a problem with brcmfmac/firmware running on OpenWrt when it comes to setting country codes. Unfortunately there is no easy workaround. Some details first: The country code being set by userspace needs a translation to a country which can be used by the firmware. This should have been done by the driver, but it isn't. The biggest problem for us (driver developers) is that the translation is board/platform specific. So the R8000 needs its own mapping which we can not determine runtime.

The only workaround for the moment is selecting the country US, but that should not be done when being in an other country.

There is also another "quick" workaround, but that will need a patch on brcmfmac and hence a new build. The country code configured by Netgear user interface gets set in NVRAM. This country code is "correct" and should not be overwritten by the driver. So by commenting out the code which which actually programs the "country" in the driver would work around the problem, but wont allow you to select a different country/region from the user interface, but once configured correctly through the netgear firmware this should not be necessary unless you physically move the AP to a different country/region.

We have had a short internal discussion on how to solve this in the future and have some ideas, but all solutions require quite a bit of work and we want to do this properly, so there wont be any good solution in the next coming days. Our main idea currently focuses on having platform data (of some sorts) which maps the country codes of cfg80211 to firmware country codes, but that requires a mapping for each platform/device. On the other hand we don't see any good alternative.  Some mapping is needed at some point to get this working.

We are sorry for the inconvenience and problems this has resulted in.
